Polygon first — Arbitrum and BSC by config.

Triviu's contracts are EVM-equivalent, so extending to another EVM chain is a configuration, not new code. Polygon is the default and reference chain; Arbitrum and BSC ship with example configs and are modelled in the simulator. Each chain travels the trilemma differently — and each carries its own Tradeoff Record in the open.

Where this actually stands

Nothing is deployed on any chain yet. v0 is pre-mainnet: read the code, run the simulator, audit the contracts. On-chain execution comes chain by chain, each only after its own audit gate clears. "Simulated" below means exactly that — a real model with real pools, no wallet and no money. It does not mean live.

Solana — a sibling, not an extension deferred

Solana is not EVM: none of these contracts port to it. Supporting Solana means a separate, ground-up build in Rust — different pool math (Orca, Raydium), a different ordering model (no public mempool; Jito bundles) — with its own audit. It is recorded as intent (Record 0006), not a config flip, and it is not available. When a build is opened, a new record supersedes the deferral and carries its own honest trilemma reading.

Why the same code travels

The contracts don't change

The Executor, Registry and Gas-Tank are EVM bytecode. A new EVM chain is a new address book (router, tokens, pools) in a config file — audited code stays audited code.

The economics don't change

The fee wall, slippage and searcher competition are the same game on every chain. Adding a chain changes the venue, not the odds — the simulator's long run shows that on each.
